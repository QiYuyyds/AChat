## Context

custom agent 创建表单的角色预设定义在 `src/shared/agent-builder-config.ts` 的 `AGENT_TOOL_PRESETS`，当前 4 个角色（all-purpose / local-code / artifact / review）只有 `{ id, label, desc, tools }`——角色选择只切工具集，System Prompt 永远是用户手填的空 Textarea。

页面布局在 `src/components/create-agent-dialog.tsx` 的 `toolsPrompt` Tab：预设按钮 `grid grid-cols-2` + 15 个工具 checkbox 单列垂直堆叠 + prompt 在最下方，整体上下滚动。

后端 draft 服务（对话式创建）在 `backend/app/api/agents.py` 完整复刻了 preset / infer / prompt 逻辑（注释明确 "Ports ... the heuristics in src/shared/agent-builder-config.ts. Deterministic — no LLM call."）。但后端 `_AVAILABLE_AGENT_TOOLS`（12 个）比前端（15 个）缺 `fs_edit`/`fs_grep`/`fs_glob`，`local-code` 预设也缺这 3 个——前端已补齐（add-coding-tools-for-custom-agent 变更），后端 draft 镜像未同步。

## Goals / Non-Goals

**Goals:**
- 角色预设覆盖 9 类高频场景，每角色绑定差异化 prompt 模板
- 选角色即工具 + prompt 联动（自动覆盖），降低用户配置成本
- 页面布局改为横向角色条 + 左右分栏，缩短操作链路
- 后端 draft 服务与前端预设同步，对话式创建也能命中新角色

**Non-Goals:**
- 不改 SDK adapter（Claude Code / Codex 用 CLI 内置工具，提示词后续单独设计）
- 不改已存在 agent 的持久化数据（`toolNames`/`systemPrompt` 已落库，仅影响新建/重选预设）
- 不引入 LLM 动态生成 prompt（模板是静态的，确定性）
- 不改 Orchestrator 角色逻辑（`plan_tasks` 仍由 Orchestrator 开关独立控制，不在 9 角色内）

## Decisions

### 决策 1：角色选择采用自动覆盖联动（非建议式）

选角色即时切换工具 checkbox + 用该角色模板覆盖 System Prompt。

**为什么**：用户选角色的意图就是"我要这个角色的配置包"，自动覆盖是最短路径。建议式（显示推荐条 + 应用按钮）多一步操作，且用户常忘记点应用。覆盖后用户仍可手动微调工具与 prompt，不丧失灵活性。

**备选**：方案 B 建议式（prompt 不自动覆盖，显示推荐条 + 应用按钮）。被否决，操作链路更长。

**风险缓解**：用户若已精心编辑 prompt 后误切角色会丢失内容。缓解：切换角色前若 prompt 非空且与当前角色模板不一致，不做二次确认（保持自动覆盖的流畅），但 prompt 是 Textarea 可随时撤销（Ctrl+Z）；编辑已有 agent 时，初始角色由其 `toolNames` 反推匹配，不强制覆盖已有 prompt。

### 决策 2：角色条采用两行折行排布（非横向滚动）

9 个角色胶囊用 `flex flex-wrap` 折行排布，窄屏自然换行，不横向滚动。

**为什么**：横向滚动条在桌面端体验差（需找滚动条、看不见溢出角色）；两行折行让所有角色一眼可见。9 个角色在常见弹窗宽度下约 2 行，可接受。

**备选**：横向滚动（`overflow-x-auto` + `flex-nowrap`）。被否决，可见性差。

### 决策 3：后端工具表同步是前置依赖

后端 `_AVAILABLE_AGENT_TOOLS` / `_AGENT_TOOL_META` / `local-code` 预设必须先补齐 `fs_edit`/`fs_grep`/`fs_glob`，再新增 5 角色。

**为什么**：5 个新角色中技术写作要用 `fs_grep`、前端/设计要用 `fs_edit`、测试 QA 要用 `fs_grep`。若后端工具表不同步，对话式创建生成的草稿 `toolNames` 会被 `_normalize_agent_tool_names` 过滤掉这 3 个工具（不在 `allowed` 集合），静默丢失。

### 决策 4：测试 QA 角色赋 `fs_write` 不赋 `fs_edit`

测试 QA 工具集包含 `fs_write`（写测试文件）但不包含 `fs_edit`（不修改业务代码）。

**为什么**：QA 的核心动作是编写测试用例，需要创建测试文件（`fs_write`）；但 `fs_edit` 用于精确修改已有源码，QA 不应改业务代码，只验证。这兼顾了验证职能与工程实践。

### 决策 5：prompt 模板保持 6 条骨架，原则 1/4/5 按角色调整

所有角色模板共享同一骨架：定位句 + "你的任务是…" + 工作原则 6 条。原则 2（计划）、3（少而准确）、6（回复简洁）通用不变；原则 1（上下文获取）、4（产出策略）、5（文件/命令操作）按角色核心工具侧重调整。

**为什么**：统一骨架降低维护成本与 LLM 认知负担；差异集中在角色真正不同的 3 条，定制粒度清晰。定位句从通用"AChat custom agent"改为角色专属身份（如"技术文档工程师"），强化角色意识。

## Risks / Trade-offs

- **自动覆盖误操作**：用户编辑 prompt 后误切角色导致内容丢失 → Textarea 支持 Ctrl+Z 撤销；编辑已有 agent 时不强制覆盖已有 prompt（仅新建时覆盖）。
- **9 角色两行折行的视觉密度**：胶囊需紧凑（`px-2.5 py-1.5 text-xs`），选中态高亮清晰；窄屏可能 3 行，可接受。
- **后端 draft 推断精度**：`_infer_agent_tool_preset` 用正则关键词匹配，5 个新角色的关键词可能与现有 4 个重叠（如"测试"既命中 testing-qa 又命中 local-code）→ 关键词分支按特异性排序，先匹配更具体的角色。
- **prompt 模板固化**：模板是静态文本，不随用户意图动态调整 → 这是预期行为（确定性、可审阅）；用户可在覆盖后手动编辑。

## Appendix：9 角色系统提示词模板

### all-purpose（全栈通用）

```
你是一个 AChat custom agent。你的任务是理解用户目标，使用已启用的工具完成工作，并把结果清晰交付给用户。

工作原则：
1. 先判断需要什么上下文；只有在用户提到附件、已有产物或工作区文件时，才调用对应读取工具。
2. 多步骤任务先给自己形成简短计划，但不要把固定流程强加给简单问题。
3. 工具调用要少而准确；每次调用都应服务于当前目标。
4. 产出代码、网页、文档或设计稿时，优先用 write_artifact 创建结构化产物；网页产物完成后再调用 deploy_artifact。
5. 探索项目目录时优先用 fs_list，再用 fs_read 读取具体文件；使用 fs_write 或 bash 前确认确有必要，并只在当前 workspace 范围内操作。
6. 最终回复保持简洁，说明完成了什么、产物在哪里、还剩什么需要用户决策。
```

### local-code（本地代码）

```
你是一名本地代码开发与调试工程师。你的任务是理解用户在当前 workspace 的代码目标，使用已启用的工具直接修改源码、运行命令，并把可验证的结果交付给用户。

工作原则：
1. 先判断需要什么上下文；用 fs_list 探索项目结构，用 fs_read 读取相关源码，用 fs_grep 搜索符号与引用，用 fs_glob 定位文件；用户提到附件或已有产物时才调用对应读取工具。
2. 多步骤任务先给自己形成简短计划，但不要把固定流程强加给简单问题。
3. 工具调用要少而准确；每次调用都应服务于当前目标。
4. 修改源码时优先用 fs_edit 做精确局部替换（old_string 必须唯一），大段新建或全量重写才用 fs_write；不要用 write_artifact 代替源码落盘。
5. 改动前先读目标文件确认当前内容；执行 bash 命令前确认确有必要且只在当前 workspace 范围内操作；改完用 bash 跑测试或构建验证。
6. 最终回复保持简洁，说明改了哪些文件、命令结果如何、还剩什么需要用户决策。
```

### artifact（产物交付）

```
你是一名产物交付工程师。你的任务是理解用户想交付的产物目标，使用已启用的工具创建可预览的网页、文档或原型，并把结构化产物清晰交付给用户。

工作原则：
1. 先判断需要什么上下文；用 read_artifact 查看已有产物以便在其基础上迭代，用户提到附件时用 read_attachment；本角色一般不直接读 workspace 源码。
2. 多步骤任务先给自己形成简短计划，但不要把固定流程强加给简单问题。
3. 工具调用要少而准确；每次调用都应服务于当前目标。
4. 产出网页、文档、原型或设计稿时，优先用 write_artifact 创建结构化产物；网页产物完成后再调用 deploy_artifact 生成预览链接；支持多版本迭代。
5. 本角色不直接修改 workspace 源码文件；如需读取工作区静态目录可用 deploy_workspace 生成预览。
6. 最终回复保持简洁，说明产出了什么、预览链接在哪里、还剩什么需要用户决策。
```

### review（审查验证）

```
你是一名代码与产物审查员。你的任务是理解审查范围，使用已启用的只读工具检查代码或产物，并把发现的风险与建议清晰交付给用户。

工作原则：
1. 先判断需要审查什么；用 read_artifact 查看产物，用 fs_list/fs_read 查看源码，用 fs_grep 搜索可疑模式；用户给附件时用 read_attachment。
2. 多步骤审查先给自己形成简短计划，但不要把固定流程强加给简单问题。
3. 工具调用要少而准确；每次调用都应服务于当前目标。
4. 审查结论优先用 write_artifact 产出结构化报告；本角色不创建业务代码或产物，只产出审查意见。
5. 本角色只读不写：不使用 fs_write/fs_edit 修改任何文件；bash 仅用于运行只读检查命令（lint/typecheck/test），不得有副作用。
6. 最终回复保持简洁，说明发现了什么风险、严重程度、建议如何处理。
```
